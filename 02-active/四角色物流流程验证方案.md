> 版本: v1.0
> 创建日期: 2026-03-25

---

## 1. 概述

设计一个完整的四角色物流流程验证方案，覆盖：API自动化测试、UI端到端测试、异常场景测试、并行测试，并支持 CI/CD 集成和手动触发。

### 1.1 四角色定义

| 角色 | Agent ID | 主要职责 |
|-----|----------|---------|
| 物流客户 | logistics-customer | 下单、确认价格、确认签收 |
| 物流调度 | logistics-dispatcher | 派发比价、选择报价、分配司机 |
| 物流网点 | logistics-branch-operator | 报价、确认收货 |
| 物流司机 | logistics-driver | 提货、配送 |

### 1.2 验证目标

- 覆盖完整订单生命周期（创建→完成）
- 验证四角色交互正确性
- 测试异常场景处理能力
- 验证并行操作稳定性
- 支持自动化 CI/CD 集成

## 4. Phase 1: 正常流程测试

### 4.1 测试步骤 (10步)

| 步骤 | 操作 | 角色 | API端点 | 预期状态变更 |
|-----|------|------|---------|------------|
| 1 | 客户下单 | customer | `POST /order` | orderId生成, status=0 |
| 2 | 调度派发比价 | dispatcher | `POST /dispatch/request-quotes` | pricingStatus=1 |
| 3 | 网点报价 | branch | `POST /network/quote` | pricingStatus=1 (报价记录) |
| 4 | 调度选择报价 | dispatcher | `POST /dispatch/select-quote` | pricingStatus=2 |
| 5 | 调度推送报价 | dispatcher | `POST /dispatch/push-quote` | status=4 |
| 6 | 客户确认价格 | customer | `POST /dispatch/confirm-price` | pricingStatus=3, status=5 |
| 7 | 调度安排提货 | dispatcher | `POST /dispatch/assign-pickup-driver` | pricingStatus=4, driverId |
| 8 | 司机提货 | driver | `POST /driver/pickup` | pricingStatus=5 |
| 9 | 网点确认收货 | branch | `POST /network/confirm-receive` | status=9 |
| 10 | 客户签收 | customer | `POST /order/sign` | status=13 |

### 4.2 验证点

- [ ] 每步API返回success=true
- [ ] 状态字段正确递增
- [ ] pricingStatus和status按预期变更
- [ ] 订单完整生命周期可追溯

## 5. Phase 2: 异常场景测试

### 5.1 测试场景

| 场景 | 触发条件 | 预期处理 | 恢复方式 |
|-----|---------|---------|---------|
| **报价超时** | 网点5分钟未报价 | 调度收到超时提醒 | 重新派发或取消 |
| **客户拒绝** | 客户拒绝价格 | 订单暂停 | 调度调整价格 |
| **司机取消** | 司机取消提货 | 订单待分配 | 调度重新分配 |
| **配送失败** | 配送过程异常 | 配送暂停 | 调度重新分配 |

### 5.2 测试步骤

#### 场景1: 报价超时

1. 客户下单 → status=0
2. 调度派发比价 → pricingStatus=1
3. 等待超时 (模拟)
4. 验证: 调度可重新派发或取消订单

#### 场景2: 客户拒绝

1. 客户下单 → 调度派发 → 网点报价 → 调度选报价
2. 调度推送报价 → status=4
3. 客户拒绝价格
4. 验证: 订单保持status=4，调度可调整

#### 场景3: 司机取消

1. 客户下单 → 确认价格 → 调度安排提货
2. 司机取消提货任务
3. 验证: pricingStatus回退，调度可重新分配

#### 场景4: 配送失败

1. 网点确认收货 → 调度分配配送司机
2. 配送过程异常
3. 验证: pricingStatus保持，调度可重新分配

## 6. Phase 3: 并行测试

### 6.1 测试场景

| 场景 | 并行数量 | 验证点 |
|-----|---------|--------|
| **多订单派发** | 5订单同时 | 状态独立、互不干扰 |
| **多网点报价** | 3网点同时 | 报价正确汇总、互不覆盖 |
| **多司机配送** | 3司机同时 | 配送状态正确、不串单 |

### 6.2 测试步骤

#### 多订单并行派发

1. customer1 创建订单A、B、C、D、E
2. dispatcher 同时派发5个订单给多个网点
3. 验证: 每个订单独立状态，不互相影响
4. 验证: pricingStatus独立追踪

#### 多网点同时报价

1. dispatcher 派发订单给 network1、network2、network3
2. 三个网点同时提交报价
3. 验证: 报价记录都保存
4. 验证: dispatcher可查看所有报价

#### 多司机同时配送

1. dispatcher 分配配送任务给 driver1、driver2、driver3
2. 三个司机同时更新配送状态
3. 验证: 每个订单状态独立更新
4. 验证: 无状态串改

## 8. 触发机制

### 8.1 CI/CD 集成 (GitHub Actions)

```yaml
name: Logistics Flow Test

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  api-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Start Backend
        run: cd backend && mvn spring-boot:run &
      - name: Run Phase 1 & 2
        run: pytest tests/phase1_phase2.py
      - name: Generate Report
        run: pytest --html=report.html

  ui-test:
    runs-on: ubuntu-latest
    needs: api-test
    steps:
      - uses: actions/checkout@v3
      - name: Start Frontend & Backend
        run: |
          cd backend && mvn spring-boot:run &
          cd vue && npm run serve &
      - name: Run Phase 4
        run: node tests/puppeteer-runner.js
```

### 8.2 手动触发

```bash
# 运行所有测试
./run-tests.sh --all

# 仅API测试
./run-tests.sh --phase 1 --phase 2

# 仅UI测试
./run-tests.sh --phase 4

# 指定场景
./run-tests.sh --scenario parallel
```

## 13. Phase 5: 多账号切换与保密性验证

### 13.1 测试目标

- 验证同一角色下多账号之间的数据隔离
- 确保各客户间订单保密性
- 验证账号切换后权限正确性
- 测试跨账号操作的安全性

### 13.2 测试账号矩阵

| 角色 | 账号1 | 账号2 | 账号3 |
|-----|-------|-------|-------|
| 客户 | customer1 | customer2 | customer3 |
| 调度 | admin1 | admin2 | - |
| 网点 | network1 | network2 | network3 |
| 司机 | driver1 | driver2 | driver3 |

### 13.3 测试场景

#### 场景1: 客户间数据隔离

| 步骤 | 操作 | 账号 | 验证点 |
|-----|------|------|--------|
| 1 | customer1 下单 | customer1 | 订单A生成，绑定customer1 |
| 2 | customer2 下单 | customer2 | 订单B生成，绑定customer2 |
| 3 | customer1 查看订单 | customer1 | 仅显示订单A，不显示订单B |
| 4 | customer2 查看订单 | customer2 | 仅显示订单B，不显示订单A |
| 5 | customer1 尝试访问订单B | customer1 | 拒绝访问，返回空或错误 |

**验证点**:
- [ ] customer1 无法看到 customer2 的订单
- [ ] customer2 无法看到 customer1 的订单
- [ ] API返回空列表或错误，而非错误数据

#### 场景2: 调度多账号协同

| 步骤 | 操作 | 账号 | 验证点 |
|-----|------|------|--------|
| 1 | admin1 派发订单A | admin1 | 订单A由admin1处理 |
| 2 | admin2 派发订单B | admin2 | 订单B由admin2处理 |
| 3 | admin1 查看待处理 | admin1 | 仅显示订单A及自己的历史 |
| 4 | admin2 查看待处理 | admin2 | 仅显示订单B及自己的历史 |
| 5 | admin1 查看全部订单 | admin1 | 可查看所有订单（管理员权限） |

**验证点**:
- [ ] admin1/admin2 处理订单独立追踪
- [ ] 管理员可查看所有订单
- [ ] 操作日志记录到对应账号

#### 场景3: 网点间数据隔离

| 步骤 | 操作 | 账号 | 验证点 |
|-----|------|------|--------|
| 1 | network1 提交订单A报价 | network1 | 报价记录绑定network1 |
| 2 | network2 提交订单A报价 | network2 | 报价记录绑定network2 |
| 3 | network1 查看报价 | network1 | 仅显示自己提交的报价 |
| 4 | 验证报价独立性 | - | 两网点报价互不覆盖 |

**验证点**:
- [ ] network1 无法修改 network2 的报价
- [ ] 每个网点仅能看到自己的报价记录
- [ ] 调度可看到所有网点报价

#### 场景4: 司机任务隔离

| 步骤 | 操作 | 账号 | 验证点 |
|-----|------|------|--------|
| 1 | 调度分配订单A给driver1 | admin | driverId=driver1 |
| 2 | 调度分配订单B给driver2 | admin | driverId=driver2 |
| 3 | driver1 查看任务 | driver1 | 仅显示订单A |
| 4 | driver2 查看任务 | driver2 | 仅显示订单B |
| 5 | driver1 尝试修改订单B | driver1 | 拒绝操作 |

**验证点**:
- [ ] driver1 仅能看到分配给自己的订单
- [ ] driver2 仅能看到分配给自己的订单
- [ ] 跨司机操作被拒绝

#### 场景5: 账号切换权限验证

| 步骤 | 操作 | 账号 | 验证点 |
|-----|------|------|--------|
| 1 | customer1 登录 | customer1 | 获取customer1的session |
| 2 | 切换到customer2 | - | session更新为customer2 |
| 3 | customer2 操作 | customer2 | 使用customer2权限 |
| 4 | 验证权限边界 | - | customer2无法执行customer1的操作 |
| 5 | 切换回customer1 | customer1 | 恢复customer1权限 |

**验证点**:
- [ ] session正确更新
- [ ] 权限边界清晰
- [ ] 切换过程无数据泄露

### 13.4 API保密性验证

| API | 验证点 | 预期行为 |
|-----|--------|---------|
| GET /order/list | userId参数隔离 | 仅返回该userId的订单 |
