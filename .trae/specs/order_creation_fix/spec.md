# 订单创建功能修复 - 产品需求文档

## Overview
- **Summary**: 修复订单创建功能中的三个问题：订单无法保存、无法显示已有订单记录、收件人显示为乱码。
- **Purpose**: 确保订单创建功能能够正常工作，包括保存订单、显示订单列表以及正确显示中文内容。
- **Target Users**: 系统管理员和业务用户。

## Goals
- 修复订单保存功能，确保订单能够成功保存到数据库。
- 修复订单列表显示功能，确保能够显示已有订单记录。
- 修复中文显示乱码问题，确保收件人等中文信息能够正确显示。

## Non-Goals (Out of Scope)
- 不修改订单创建功能的基本流程和UI设计。
- 不添加新的订单相关功能。
- 不修改其他模块的功能。

## Background & Context
- 系统使用Spring Boot 2.7.15 + MyBatis-Plus 3.5.3.1 + MySQL 8.0 + Vue 3 + Element Plus构建。
- 订单创建功能已经实现了业务用户选择、发件人信息自动填充、收件人信息筛选等功能。
- 目前存在三个问题：订单无法保存、无法显示已有订单记录、收件人显示为乱码。

## Functional Requirements
- **FR-1**: 订单保存功能，确保订单能够成功保存到数据库。
- **FR-2**: 订单列表显示功能，确保能够显示已有订单记录。
- **FR-3**: 中文显示功能，确保收件人等中文信息能够正确显示。

## Non-Functional Requirements
- **NFR-1**: 性能要求，订单保存和查询操作响应时间不超过1秒。
- **NFR-2**: 可靠性要求，订单数据保存成功率达到100%。
- **NFR-3**: 兼容性要求，确保在不同浏览器和设备上都能正常显示。

## Constraints
- **Technical**: 使用现有的技术栈，不引入新的框架或库。
- **Business**: 修复工作需要在不影响现有功能的情况下完成。
- **Dependencies**: 依赖MySQL数据库和后端API服务。

## Assumptions
- 后端服务已经正常运行。
- 数据库连接正常。
- 前端代码逻辑正确，问题出在后端或数据库层面。

## Acceptance Criteria

### AC-1: 订单保存功能
- **Given**: 用户填写完整的订单信息并点击保存按钮。
- **When**: 系统调用订单保存API。
- **Then**: 订单成功保存到数据库，返回成功状态。
- **Verification**: `programmatic`
- **Notes**: 确保订单表结构包含所有必要的字段。

### AC-2: 订单列表显示功能
- **Given**: 用户进入订单管理页面。
- **When**: 系统调用订单列表API。
- **Then**: 页面显示所有已保存的订单记录。
- **Verification**: `programmatic`
- **Notes**: 确保订单表中有数据，且API能够正确返回。

### AC-3: 中文显示功能
- **Given**: 订单中包含中文信息（如发件人、收件人姓名和地址）。
- **When**: 系统显示订单信息。
- **Then**: 中文信息显示为正常汉字，无乱码。
- **Verification**: `human-judgment`
- **Notes**: 确保数据库字符集设置正确，支持中文。
