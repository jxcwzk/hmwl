# 业务客户收件人信息返回问题修复

## 1. 简介

### 1.1 文档目的
本文档定义了修复业务客户表中收件人信息返回空的问题的需求和实现方案，确保在创建订单时能够正确从业务客户表中选择收件人信息。

### 1.2 术语定义
| 术语 | 解释 |
|------|------|
| 业务用户 | 发起物流订单的企业用户 |
| 业务客户 | 存储在业务客户表中的发件人或收件人信息 |
| 发件人信息 | 业务客户表中 type=0 的记录 |
| 收件人信息 | 业务客户表中 type=1 的记录 |

## 2. 问题分析

### 2.1 问题描述
- 发件人信息能够正常从业务客户表中返回
- 收件人信息返回空，无法在订单页面选择
- 前端调用 API 路径正确：`/api/business-customer/listByBusinessUserIdAndType/{businessUserId}/1`
- 后端 API 实现逻辑正确

### 2.2 可能原因
1. 数据库中业务客户表缺少 type=1 的记录
2. 业务客户表的 type 字段可能没有正确设置
3. 数据插入时 type 字段默认值问题

## 3. 功能需求

### 3.1 核心功能
1. **业务客户类型管理**：确保业务客户表中的记录正确设置 type 字段
2. **收件人信息查询**：确保 API 能够正确返回 type=1 的业务客户记录
3. **前端显示优化**：确保订单页面能够正确显示和选择收件人信息

### 3.2 功能详情
| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 业务客户表结构检查 | 检查业务客户表的结构，确保 type 字段存在且类型正确 | 高 |
| 数据检查 | 检查业务客户表中是否存在 type=1 的记录 | 高 |
| API 测试 | 测试 `/api/business-customer/listByBusinessUserIdAndType/{businessUserId}/1` API 是否返回数据 | 高 |
| 前端调用优化 | 确保前端正确调用 API 并处理返回数据 | 中 |
| 业务客户添加功能 | 确保添加业务客户时能够正确设置 type 字段 | 中 |

## 4. 非功能需求

### 4.1 性能需求
- API 响应时间不超过 500ms
- 前端页面加载时间不超过 2s

### 4.2 可靠性需求
- 数据查询成功率达到 99.9%
- 系统能够处理并发请求

### 4.3 安全性需求
- API 调用需要进行权限验证
- 数据传输采用 HTTPS 加密

## 5. 数据需求

### 5.1 数据结构
| 表名 | 字段 | 类型 | 描述 |
|------|------|------|------|
| business_customer | id | BIGINT | 主键 |
| business_customer | customer_name | VARCHAR | 客户名称 |
| business_customer | contact | VARCHAR | 联系人 |
| business_customer | address | VARCHAR | 地址 |
| business_customer | remark | VARCHAR | 备注 |
| business_customer | business_user_id | BIGINT | 业务用户ID |
| business_customer | type | INT | 类型，0表示发件人信息，1表示收件人信息 |
| business_customer | create_time | DATETIME | 创建时间 |
| business_customer | update_time | DATETIME | 更新时间 |

### 5.2 数据迁移

- 如缺少 type 字段，需要添加并设置默认值
