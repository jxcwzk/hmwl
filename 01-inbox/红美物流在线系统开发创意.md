## 1. 图片上传大小限制与压缩功能

### Status
- ✅ 已完成（2026-03-12）

### 原始想法
> 由于使用的腾讯云存储，上传的图片在进行预览时候会产生额外的cdn费用，因此我需要严格限制上传图片的大小，帮我想个办法，如何设置一个条件，图片大小低于3Mb可以直接上传，大于3MB的需要转换为小于3MB的小格式图片上传到服务器

### 详细描述
- 在图片上传到腾讯云COS之前进行大小判断
- 图片小于2MB：直接上传原图
- 图片大于等于2MB：自动压缩至小于2MB后再上传
- 压缩质量：70%
- 支持多种图片格式（jpg、png、gif、bmp、webp）

### 相关想法
- 与现有的 SenderImageService 和 ReceiptImageService 相关
- 涉及腾讯云COS存储费用优化

### 优先级评估
- 高优先级
- 直接影响CDN流量费用控制

### 备注
- 功能已实现，使用 Thumbnailator 库进行图片压缩
- 已完成 SenderImageServiceImpl 和 ReceiptImageServiceImpl 的上传逻辑修改

## 2. 多角色订单管理系统需求

### 原始想法
> 我需要目前的三个角色各有一个单独的界面，就是管理员拥有查看所有订单的权限，无需自动筛选，并且作为管理员，在点击订单详情时，可以指派订单给到司机，这样，作为司机，就能看到指派给自己的订单，另外，作为管理员，还需要有估价和指派网点的功能，这样，当估价最终确认后，客户端就能看到预估价，另外，我还要再增加一个角色，就是网点，网点界面的特色点就是，它能够更新物流的进度，这样，客户在查看其自己的订单时候，就可以看到物流的进展，另外，就是作为网点，可以对于这个订单的价格做最终确认，顺序就是：网点提供报价--》红美物流获得底价，乘以一个系数作为给业务用户的报价，但是这个价格是可以手动改的。

### 详细描述

#### 管理员界面
- 查看所有订单，无需自动筛选
- 点击订单详情时可以指派订单给司机
- 有估价和指派网点的功能
- 获得网点提供的底价，乘以系数作为给业务用户的报价，可手动修改

#### 司机界面
- 看到指派给自己的订单

#### 客户界面
- 看到自己的订单
- 看到物流进展
- 看到预估价

#### 新增网点角色
- 能够更新物流进度
- 对于订单价格做最终确认
- 提供报价给红美物流

### 相关想法
- 与订单模块关联
- 与物流模块关联
- 与用户角色管理关联
- 与价格计算模块关联

### 优先级评估
- 高优先级
- 理由：涉及核心业务流程，影响多个角色的使用体验，是系统的重要功能

### 备注
- 价格计算流程：网点报价 → 红美物流获得底价 → 乘以系数 → 给业务用户报价（可手动修改）
- 物流进度更新：网点更新 → 客户查看
- 订单指派：管理员指派 → 司机查看

## 3. 小程序/VUE/后端云服务三者打通需求

### Status
- ✅ 已完成 (2026-03-13)

### 原始想法
> 我现在需要重新调整已有的miniprogram文件夹的内容，基于现有的后端接口，目标是让微信小程序，前端vue界面，后端云服务三者打通

### 详细描述

#### 当前状态
- **微信小程序**: 使用微信云开发（wx.cloud），有独立云函数，大部分页面使用模拟数据
- **前端Vue**: 使用axios请求后端Java API
- **后端**: Spring Boot Java服务
- **图片存储**: 开发阶段的图片类文档信息都上传到腾讯云COS

#### 目标
- 小程序改为调用后端Java API（和Vue前端一致）
- 三端数据源统一
- 保留云开发功能（图片上传等）
- 后续数据迁移到腾讯云数据库

#### 技术方案
1. **API调用**: 从wx.cloud改为wx.request请求后端Java API
2. **登录机制**: 
   - 小程序端微信自动登录
   - 登录后由红美物流后台管理员指派身份信息
   - 指派身份后才能调用相关功能
3. **优先对接**: 订单界面
4. **部署策略**: 
   - 开发阶段: baseURL为本地
   - 正式发布: 使用腾讯云数据库

### 优先级评估
- 高优先级
- 系统架构重大调整，订单模块优先

### 备注
- 需要项目管理系统推进

- 保留核心寄件功能

- 与网页端 Order.vue 页面保持一致
- 复用现有的货物照片上传接口
- 需要获取 token 和 businessUserId 进行身份验证

## 5. 订单创建页面必填项及发收件人筛选功能

> **状态：已完成** ✅

### 需求/PRD状态
- **需求状态**：已确认 ✅
- **PRD状态**：已评审 ✅

### 原始想法
> 前端页面需要设置一些项目为必填项，防止重要物流信息遗漏，业务用户的最为重要，因为他是物流订单的来源，后面的发件人信息和收件人信息和他有关系，需要基于业务用户信息，筛选发件人信息和收件人信息，因为这俩都是跟对应的业务用户有直接联系，不同业务用户的发件人信息和收件人信息不一样

### 详细描述
- **必填项设置**：前端订单创建页面需要将一些重要字段设为必填，防止重要物流信息遗漏
- **业务用户重要性**：业务用户是物流订单的来源，是最重要的关联实体
- **发件人/收件人筛选**：
  - 发件人信息和收件人信息需要与业务用户关联
  - 选择不同业务用户时，发件人和收件人下拉列表应显示该业务用户关联的对应人员
  - 不同业务用户拥有不同的发件人和收件人资源

### 相关想法
- 与现有 BusinessCustomer（业务客户）、BusinessRecipient（业务收件人）模块关联
- 需要后端提供根据业务用户ID筛选发件人/收件人的API
- 可能需要调整订单创建页面的交互逻辑，先选择业务用户，再选择发件人/收件人

### 优先级评估
- 高优先级
- 理由：直接影响订单数据完整性，防止重要信息遗漏，是业务流程的核心环节

### 备注
- 初步想法，需要进一步细化必填项具体有哪些字段

## 7. 订单距离自动计算功能

> **状态：已完成** ✅

### 需求/PRD状态
- **需求状态**：已完成 ✅
- **备注**：后端距离计算服务已实现，前端表单包含距离字段

### 原始想法
> 我想给订单页面的距离加上一个距离算法，根据发货地址和收货地址，自动计算大致的距离，不用手动填写距离，方便后续价格计算

### 详细描述
- 在创建/编辑订单时，根据发货地址和收货地址自动计算两点之间的距离
- 距离计算后自动填充到距离字段，无需手动填写
- 距离数据用于后续的价格计算（运费 = 距离 × 单价）
- 可以使用地图 API（如腾讯地图、高德地图）或直线距离算法

### 相关想法
- 与订单模块关联
- 需要获取发货网点和收货网点的地址信息
- 可能需要调用地图服务的距离计算 API
- 影响价格计算模块

### 优先级评估
- 高优先级
- 自动化数据录入，减少人工错误
- 是价格计算的前置功能

### 备注
- 需要选择合适的距离计算方案
- 考虑离线场景的处理方式
- 需要确认地图 API 的配置

## 9. 财务结算界面功能需求

### Status
- ✅ 已完成（2026-03-12） - 已实现订单联动、金额计算、筛选、开票功能

### 原始想法
> 关于财务结算界面，我需要和订单界面联动，当订单界面的订单完成支付后，会在财务结算界面增加记录，然后可以有筛选的功能，便于按照业务用户和时间段筛选记录，最终还要有一键开发票功能

### 详细描述
- **订单联动**：当订单界面完成支付或回单拍照上传后，自动在财务结算界面增加一条结算记录
  - 触发条件1：订单完成支付（payment_status = paid）
  - 触发条件2：订单完成（回单拍照上传，order_status = completed）
- **金额计算**：
  - 订单原金额：订单界面显示的支付给网点的金额（网点成本）
  - 推荐客户价：基于利润率30%计算，即 `客户报价 = 网点成本 / (1 - 30%) = 网点成本 × 1.4286`
  - 最终金额：推荐值为默认值，允许财务人员手动输入调整确认
- **筛选功能**：支持按业务用户（customer）和时间段（date range）筛选结算记录
- **一键开票**：提供快捷操作按钮，一键生成发票

### 相关想法
- 与订单模块关联，需要监听订单支付状态变化
- 可能需要新增财务记录数据表
- 开票功能可能需要与税票系统对接

### 优先级评估
- 高优先级
- 核心业务功能，财务管理是业务闭环的关键环节

### 备注
- 初步想法，需要进一步细化技术实现方案
