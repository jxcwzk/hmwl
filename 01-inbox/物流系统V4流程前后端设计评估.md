# 物流系统V4流程 - 前后端设计评估报告

## 一、评估概述

### V4流程核心变化
基于V3流程优化，新增13个任务，优化状态流转，明确调度与网点的职责边界：

```
待接单 → 已接单 → 比价中 → 比价完成 → 待确认价格 → 价格已确认 → 待提货 → 已提货 → 待入库 → 已入库 → 待配送 → 配送中 → 派送中 → 已签收
```

### 职责边界明确
- **调度**：核心中枢，负责接单、比价、价格、司机管理
- **网点**：响应比价、确认收货、仓储分拣
- **提货司机**：归调度管理，负责提货送网点
- **配送司机**：归调度管理，负责网点到客户的配送

## 四、关键问题清单

### P0问题（必须解决）

#### 1. 比价流程断裂
**问题**：调度派发比价 → 网点响应报价 → 调度选择网点 的完整流程没有实现

**影响**：
- 无法实现网点竞价机制
- 价格形成机制不透明
- 调度决策缺乏数据支持

**建议**：
- 新增 `POST /api/dispatch/request-quotes` API
- 新增 `POST /api/network/quote` API（网点报价）
- 前端新增比价请求和报价响应界面

#### 2. 司机职责未区分
**问题**：提货司机和配送司机没有明确区分，都是用driverId

**影响**：
- 提货和配送职责混淆
- 司机调度不精准
- 考核机制无法区分

**建议**：
- Order实体增加 `pickupDriverId` 和 `deliveryDriverId` 字段
- 新增相关API支持分别管理
- 前端界面区分提货和配送司机

#### 3. 网点确认收货环节缺失
**问题**：货物送达网点后没有确认收货的环节

**影响**：
- 货物交接责任不清
- 仓储管理不完善
- 异常追踪困难

**建议**：
- 新增 `POST /api/network/confirm-receive` API
- 前端新增网点确认收货界面
- 增加库存管理功能

### P1问题（应该解决）

#### 4. 状态流转不完整
**问题**：当前订单状态只有简单的0-5状态，无法支持V4流程的14个状态

**影响**：
- 流程控制不精细
- 状态判断复杂
- 追踪困难

#### 5. 价格推送机制缺失
**问题**：调度选择报价后没有推送给客户的机制

**影响**：
- 客户无法及时获知价格
- 价格确认流程不闭环
- 客户体验差

## 五、优化建议

### 后端优化

#### 1. 新增API清单
```java
// 比价流程
POST /api/dispatch/request-quotes    // 调度派发比价请求
POST /api/network/quote              // 网点响应报价
POST /api/dispatch/push-quote       // 调度推送报价给客户

// 司机管理
POST /api/dispatch/assign-pickup-driver    // 分配提货司机
POST /api/dispatch/assign-delivery-driver   // 分配配送司机

// 网点操作
POST /api/network/confirm-receive    // 网点确认收货
GET  /api/network/orders             // 网点订单列表

// 状态更新
POST /api/order/update-status-extended     // 扩展状态更新
```
